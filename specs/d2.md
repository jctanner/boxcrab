# D2 Rendering Specification

A language-agnostic specification of how D2 transforms diagram-as-code text
into rendered output. Written for implementors building a D2 renderer in Rust
or other languages.

## 1. Overview

D2 is a declarative diagram-as-code language. It uses a concise text syntax
to define shapes, connections, and styling, then compiles and lays out the
result as an SVG diagram.

Key properties:
- **Nested containers**: shapes can contain other shapes via `{ }` blocks
- **Implicit creation**: referencing a shape in a connection creates it
- **Multiple layout engines**: dagre (default), ELK, grid, sequence
- **Theme system**: 20 built-in color themes with slot-based color palettes
- **Rich shapes**: 25 shape types including special forms (class diagrams,
  SQL tables, sequence diagrams)

The fundamental rendering flow:

```
D2 Text → Parse → AST → Compile → IR → Graph
  → Layout → Export → Target Model → SVG Render
```

## 2. High-Level Pipeline

### 2.1 Pipeline Stages

```
1. Parse       — Tokenize and parse D2 text into an AST (d2ast.Map)
2. Compile IR  — Transform AST into typed IR (d2ir.Map) with
                 glob expansion, import resolution, class application
3. Compile     — Build graph model (d2graph.Graph) with objects and edges
4. Theme       — Apply color theme to graph
5. Measure     — Measure text dimensions for label sizing
6. Layout      — Run layout engine (dagre/ELK/grid/sequence)
7. Export      — Convert graph to target model (d2target.Diagram)
8. Render      — Generate SVG from target model
```

### 2.2 Data Flow

```
D2 Source Text
  │
  ▼
d2ast.Map (Abstract Syntax Tree)
  │  Keys, Edges, Values, Comments
  ▼
d2ir.Map (Intermediate Representation)
  │  Fields, Edges with references
  │  Globs expanded, imports resolved, classes applied
  ▼
d2graph.Graph
  │  Objects (shapes with geometry)
  │  Edges (connections with routing)
  │  Layers / Scenarios / Steps (sub-boards)
  ▼
d2target.Diagram
  │  Shapes (positioned, styled)
  │  Connections (routed, styled)
  ▼
SVG Output
```

## 3. Lexical Analysis

### 3.1 Character Set

D2 files are UTF-8 text.

### 3.2 Comments

```
# Single-line comment (# at start of token position)
# Comments can chain across consecutive lines starting with #

"""
Block comment — multi-line
"""
```

### 3.3 String Types

**Unquoted strings**: default, terminated by special characters.
```
hello world            # simple label
```

**Double-quoted strings**: support escape sequences and substitutions.
```
"hello \"world\""
"value is ${var.name}"
```

**Single-quoted strings**: literal, no escaping.
```
'no escaping here'
```

**Block strings**: multi-line with optional language tag.
```
|md
  # Markdown Title
  Body text with **bold**
|

|sql
  SELECT * FROM users
|

||||
  Content with | pipes
||||
```

Block strings use `|` delimiters. Extra `|` characters allow content
containing `|`. The language tag (md, sql, etc.) determines rendering.

### 3.4 Escape Sequences

In double-quoted and unquoted strings:
`\n` `\t` `\r` `\\` `\"` `\a` `\b` `\f` `\v`

Line continuation: `\` at end of line joins the next line.

### 3.5 Substitutions

Variable interpolation in unquoted and double-quoted strings:
```
${path.to.variable}
```

Resolved during compilation from the `vars` block.

## 4. Grammar and Parsing

### 4.1 Top-Level Grammar

```
file       = map
map        = { map_node }*
map_node   = comment | block_comment | key_stmt | substitution | import

key_stmt   = [& | !&] (key_path | edge_group) [: value | { map }]

key_path   = identifier { . identifier }*
identifier = unquoted_string | double_quoted_string | single_quoted_string

edge_group = ( edges )
edges      = key_path { edge key_path }*
edge       = [< | *] -- [> | * | ε]
             # e.g.: ->, <-, <->, --, *->
```

### 4.2 Value Types

```
value      = scalar | array | map | import
scalar     = null | boolean | number | string
array      = [ { array_node [;] }* ]
map        = { { map_node [;] }* }
```

### 4.3 Key Path (Dot Notation)

Shapes are addressed via dot-separated paths:
```
a.b.c             # nested: a contains b contains c
a."quoted key".c  # quoted segments for special characters
```

### 4.4 Statement Separators

Statements are separated by newlines or semicolons (`;`). Semicolons
allow multiple statements on one line:
```
a; b; a -> b
```

### 4.5 Filter Prefixes

```
&key      # Apply only if key already exists (ampersand filter)
!&key     # Apply only if key does NOT exist (negated filter)
```

Used with globs for conditional styling.

## 5. AST Structure

### 5.1 Core Node Types

```
Map
  nodes: Vec<MapNode>

MapNode = Comment | BlockComment | Key | Substitution | Import

Key
  key:    Option<KeyPath>        // shape path (None for edge-only)
  edges:  Vec<Edge>              // connections in this statement
  value:  Option<Value>          // label, style block, etc.

KeyPath
  path:   Vec<StringNode>       // dot-separated segments

Edge
  src:    KeyPath
  dst:    KeyPath
  src_arrow: Option<ArrowChar>  // <, *
  dst_arrow: Option<ArrowChar>  // >, *
```

### 5.2 Scalar Types

```
Null                           // null
Boolean(bool)                  // true, false
Number(BigRational)            // arbitrary precision
UnquotedString(Vec<Part>)      // parts may include Substitutions
DoubleQuotedString(Vec<Part>)  // with escapes and substitutions
SingleQuotedString(String)     // literal
BlockString { tag, content }   // |tag ... |
```

### 5.3 Substitution and Import

```
Substitution
  path: KeyPath               // ${path.to.var}

Import
  path: String                // ...@./path/to/file.d2
  # or
  var_path: KeyPath           // ...$var.name
```

## 6. Intermediate Representation

### 6.1 IR Structure

The IR adds strong typing and reference tracking:

```
IR Map
  fields: Vec<Field>
  edges:  Vec<Edge>

Field
  name:       String
  primary:    Option<Scalar>     // scalar value
  composite:  Option<Composite>  // array or nested map
  references: Vec<FieldRef>      // all assignment sites

Edge
  id:         EdgeID             // (src_path, dst_path, index)
  primary:    Option<Scalar>     // label
  map:        Option<Map>        // edge properties
  references: Vec<EdgeRef>
```

### 6.2 IR Compilation Steps

```
1. Walk AST recursively
2. Merge duplicate field assignments (last wins for scalars)
3. Expand glob patterns (* and **) to matching fields
4. Resolve imports (...@path)
5. Apply class definitions to matching objects
6. Resolve variable substitutions (${...})
7. Track all reference sites for error reporting
```

## 7. Graph Model

### 7.1 Graph Structure

```
Graph
  root:       Object             // top-level container
  objects:    Vec<Object>        // all shapes (flat list)
  edges:      Vec<Edge>          // all connections
  layers:     Vec<Graph>         // layer sub-boards
  scenarios:  Vec<Graph>         // scenario sub-boards
  steps:      Vec<Graph>         // step sub-boards
  theme:      Theme              // applied color theme
```

### 7.2 Object (Shape)

```
Object
  id:           String           // full D2 key path
  parent:       Option<Object>   // parent container
  children:     HashMap<String, Object>
  children_array: Vec<Object>    // ordered

  // Geometry (set by layout)
  box:          Box { top_left: Point, width: f64, height: f64 }

  // Attributes
  label:        String
  shape:        String           // shape type name
  icon:         Option<URL>
  tooltip:      Option<String>
  link:         Option<String>
  style:        Style
  classes:      Vec<String>

  // Label/Icon positioning
  label_position: Option<String>
  icon_position:  Option<String>

  // User-specified dimensions
  width_attr:   Option<i64>
  height_attr:  Option<i64>
  top:          Option<i64>
  left:         Option<i64>

  // Grid layout
  grid_rows:    Option<i64>
  grid_columns: Option<i64>
  grid_gap:     Option<i64>
  vertical_gap: Option<i64>
  horizontal_gap: Option<i64>

  // Direction override
  direction:    Option<String>   // up, down, left, right

  // Special shape data
  class:        Option<ClassDef>
  sql_table:    Option<SQLTableDef>

  z_index:      i32
```

### 7.3 Edge (Connection)

```
Edge
  index:        i32
  src:          Object
  dst:          Object
  src_arrow:    bool
  dst_arrow:    bool
  src_arrowhead: Option<Attributes>
  dst_arrowhead: Option<Attributes>

  // Geometry (set by layout)
  route:        Vec<Point>
  is_curve:     bool

  // Attributes
  label:        Option<String>
  style:        Style
  label_position: Option<String>
  label_percentage: Option<f64>  // 0.0-1.0 along route

  z_index:      i32
```

## 9. Connections and Arrowheads

### 9.1 Arrow Syntax

| Syntax | Source Arrow | Dest Arrow | Description |
|--------|-------------|-----------|-------------|
| `->` | none | triangle | Forward arrow |
| `<-` | triangle | none | Reverse arrow |
| `<->` | triangle | triangle | Bidirectional |
| `--` | none | none | Undirected line |

### 9.2 Connection Declaration

```d2
# Simple connection
a -> b

# Connection with label
a -> b: sends data

# Connection with properties
a -> b: {
  label: sends data
  style.stroke: red
  style.animated: true
  target-arrowhead: {
    shape: diamond
  }
}

# Chained connections
a -> b -> c -> d

# Edge group
(a -> b -> c): { style.stroke: blue }
```

### 9.3 Arrowhead Types (19)

| Arrowhead | Keyword | Description |
|-----------|---------|-------------|
| None | `none` | No arrowhead |
| Arrow | `arrow` | Chevron (open) |
| Triangle | `triangle` | Filled triangle (default) |
| Unfilled Triangle | `unfilled-triangle` | Triangle outline |
| Diamond | `diamond` | Diamond outline |
| Filled Diamond | `filled-diamond` | Filled diamond |
| Circle | `circle` | Circle outline |
| Filled Circle | `filled-circle` | Filled circle |
| Cross | `cross` | X/plus shape |
| Box | `box` | Square outline |
| Filled Box | `filled-box` | Filled square |
| Line | `line` | Perpendicular line (fat arrow) |
| CF One | `cf-one` | Crow's foot: one |
| CF Many | `cf-many` | Crow's foot: many |
| CF One Required | `cf-one-required` | Crow's foot: required one |
| CF Many Required | `cf-many-required` | Crow's foot: required many |

### 9.4 Arrowhead Configuration

```d2
a -> b: {
  source-arrowhead: {
    shape: circle
    label: "1"
  }
  target-arrowhead: {
    shape: cf-many
    label: "*"
    style.filled: true
  }
}
```

### 9.5 Edge Indexing

Multiple edges between the same pair are indexed:

```d2
a -> b: first
a -> b: second

# Reference specific edge
(a -> b)[0].style.stroke: red    # first edge
(a -> b)[1].style.stroke: blue   # second edge
(a -> b)[*].style.animated: true # all edges
```

### 10.3 Classes

Reusable style sets:

```d2
classes: {
  error: {
    style: {
      fill: "#ffcccc"
      stroke: red
      font-color: red
    }
  }
  success: {
    style.fill: "#ccffcc"
  }
}

box1: { class: error }
box2: { class: success }
```

### 10.4 Glob Patterns

Apply styles to multiple shapes:

```d2
# All top-level shapes
*.style.fill: "#eee"

# All shapes at any depth
**.style.stroke-width: 2

# Filtered: only containers
&*.style.fill: "#f0f0f0"

# Filtered: only non-containers
!&*.style.fill: "#ffffff"
```

## 11. Containers and Nesting

### 11.1 Container Creation

Any shape with children becomes a container:

```d2
server: {
  web: Web Server
  api: API Server
  db: Database { shape: cylinder }

  web -> api -> db
}
```

### 11.2 Container Types

| Type | Condition | Layout |
|------|-----------|--------|
| Regular | Has children | Hierarchical (dagre/ELK) |
| Grid | Has `grid-rows` or `grid-columns` | Grid |
| Sequence | `shape: sequence_diagram` | Sequence |
| Hierarchy | `shape: hierarchy` | Tree |

### 11.3 Container Rendering

Containers render as their shape type with children positioned inside:

```
╭─── Server ─────────────────────────────╮
│                                        │
│  ┌──────────┐   ┌──────────┐   ╭────╮  │
│  │ Web      │──→│ API      │──→│ DB │  │
│  │ Server   │   │ Server   │   ╰────╯  │
│  └──────────┘   └──────────┘           │
│                                        │
╰────────────────────────────────────────╯
```

The container label appears at the top, and children are laid out
within the container's bounds with padding.

### 11.4 Cross-Container Connections

Dot notation references shapes inside containers:

```d2
frontend.app -> backend.api: REST
backend.api -> backend.db: SQL
```

Edges can cross container boundaries. The layout engine routes them
through container borders.

### 11.5 Grid Diagrams

```d2
grid: {
  grid-rows: 2
  grid-columns: 3
  grid-gap: 10

  a; b; c; d; e; f
}
```

Children are positioned in a grid with the specified rows/columns.

## 12. Special Diagram Types

### 12.1 Class Diagrams

```d2
MyClass: {
  shape: class

  +publicField: string
  -privateField: int
  #protectedField: bool

  +publicMethod(arg: string): void
  -privateMethod(): int
}
```

Rendered as a sectioned box:

```
╭────────────────────╮
│     MyClass        │  ← class name (bold, mono)
├────────────────────┤
│ +publicField: str  │  ← fields section
│ -privateField: int │
│ #protectedField    │
├────────────────────┤
│ +publicMethod()    │  ← methods section
│ -privateMethod()   │
╰────────────────────╯
```

Visibility markers: `+` public, `-` private, `#` protected.

### 12.2 SQL Tables

```d2
users: {
  shape: sql_table
  id: int { constraint: primary_key }
  name: varchar
  email: varchar { constraint: unique }
  org_id: int { constraint: foreign_key }
}
```

Rendered as a table with columns:

```
╭────────────────────────────────╮
│            users               │  ← table name
├───────────┬──────────┬─────────┤
│ id        │ int      │ PK      │
│ name      │ varchar  │         │
│ email     │ varchar  │ UQ      │
│ org_id    │ int      │ FK      │
╰───────────┴──────────┴─────────╯
```

Edges can connect to specific columns via SQL table column indices.

### 12.3 Sequence Diagrams

```d2
shape: sequence_diagram

alice: Alice
bob: Bob

alice -> bob: Hello
bob -> alice: Hi
alice -> bob: How are you?
```

Rendered with actors on horizontal axis and messages as vertical arrows:

```
  ┌───────┐                ┌─────┐
  │ Alice │                │ Bob │
  └───┬───┘                └──┬──┘
      │                       │
      │──── Hello ───────────→│
      │                       │
      │←──── Hi ──────────────│
      │                       │
      │── How are you? ──────→│
      │                       │
```

### 12.4 Code Blocks

```d2
snippet: |go
  func main() {
      fmt.Println("Hello")
  }
|
snippet.shape: code
```

Rendered with syntax highlighting via the Chroma library. Each token
gets color/weight from the language grammar.

### 12.5 Markdown Content

```d2
readme: |md
  # Title
  This is **bold** and *italic*.
  - List item 1
  - List item 2
|
```

Rendered via `<foreignObject>` in SVG with HTML content.

### 12.6 Images

```d2
logo: {
  shape: image
  icon: https://example.com/logo.png
}
```

## 13. Layout

### 13.1 Layout Engines

| Engine | Type | Best For |
|--------|------|----------|
| dagre | Hierarchical (Sugiyama) | General directed graphs (default) |
| ELK | Hierarchical + force | Complex nested graphs |
| Grid | Row/column | Tabular arrangements |
| Sequence | Timeline | Sequence diagrams |
| Near | Absolute | Fixed-position elements |

### 13.2 Direction

```d2
direction: right    # default
direction: down
direction: left
direction: up
```

Containers can override direction for their children.

### 13.3 Layout Pipeline

```
1. Determine diagram type per container:
   - grid-rows/grid-columns → grid layout
   - sequence_diagram → sequence layout
   - has near attribute → near positioning
   - default → dagre/ELK

2. Layout nested containers (depth-first):
   a. Extract nested sub-graphs
   b. Recursively layout each sub-graph
   c. Size the container to fit its children + padding
   d. Return to parent for parent-level layout

3. Run top-level layout:
   a. Assign ranks (layers) based on edge direction
   b. Order nodes within ranks to minimize crossings
   c. Assign coordinates respecting separation constraints
   d. Route edges between connection points

4. Apply constant-near positioning:
   - Elements with near: top-left, etc. are placed at
     fixed positions relative to diagram bounds

5. Compute final bounding box
```

### 13.4 Dagre Layout Parameters

```
node_separation:  60px (horizontal between nodes)
edge_separation:  varies
rank_separation:  60px (vertical between layers)
```

### 13.5 Edge Routing

Edges are routed as either curves or polylines:

- **Curved** (`is_curve: true`): Cubic Bézier splines through waypoints
- **Straight with corners** (`border_radius > 0`): Polyline with
  rounded corners at each bend point
- **Straight** (default): Polyline through waypoints

## 14. Theme System

### 14.1 Color Palette Structure

Every theme defines 13 color slots:

```
Neutrals (7 slots):
  N1  — Darkest (primary text, strokes)
  N2  — Secondary text, connection labels
  N3  — Tertiary elements
  N4  — Borders, dividers
  N5  — Subtle borders
  N6  — Subtle backgrounds
  N7  — Background (lightest)

Base Colors (6 slots):
  B1  — Darkest accent (primary container stroke)
  B2  — Primary accent (container fill)
  B3  — Medium accent (container outline)
  B4  — Light accent (inner element fill)
  B5  — Lighter accent
  B6  — Lightest accent (default shape fill)

Alternative Colors (4 slots):
  AA2 — Alternative accent A
  AA4 — Light alternative A
  AA5 — Lightest alternative A
  AB4 — Alternative accent B
  AB5 — Light alternative B
```

## 15. Shape Rendering

### 15.1 SVG Output Structure

```xml
<svg xmlns="http://www.w3.org/2000/svg"
     viewBox="x y width height"
     width="..." height="...">

  <defs>
    <!-- Shadow filter -->
    <filter id="shadow-filter">
      <feGaussianBlur stdDeviation="1.7" />
      <feFlood flood-color="#3d4574" flood-opacity="0.4" />
      <feComposite operator="in" />
      <feOffset dx="3" dy="5" />
      <feBlend mode="normal" />
    </filter>

    <!-- Fill patterns (dots, lines, grain, paper) -->
    <pattern id="dots-..." width="15" height="15">
      <!-- 9 small rectangles, opacity 0.1, mix-blend-mode: multiply -->
    </pattern>

    <!-- Arrowhead markers -->
    <marker id="mk-..." ...>
      <polygon points="..." />
    </marker>
  </defs>

  <!-- Shapes -->
  <g id="shape-id">
    <rect x="..." y="..." width="..." height="..."
          rx="..." fill="..." stroke="..." stroke-width="..." />
    <text x="..." y="..." font-size="..." fill="...">
      <tspan>Label Line 1</tspan>
      <tspan dy="...">Label Line 2</tspan>
    </text>
  </g>

  <!-- Connections -->
  <g id="conn-id">
    <path d="M... C..." stroke="..." stroke-width="..."
          marker-end="url(#mk-...)" />
    <text>Label</text>
  </g>
</svg>
```

### 15.2 Rectangle Rendering

```xml
<rect x="{tl.x}" y="{tl.y}"
      width="{w}" height="{h}"
      rx="{border_radius}"
      fill="{fill}" stroke="{stroke}"
      stroke-width="{stroke_width}"
      opacity="{opacity}" />
```

If `fill-pattern` is set, add a second rect with the pattern:
```xml
<rect ... fill="url(#pattern-id)" />
```

### 15.3 Oval / Circle Rendering

```xml
<ellipse cx="{tl.x + w/2}" cy="{tl.y + h/2}"
         rx="{w/2}" ry="{h/2}"
         fill="..." stroke="..." />
```

Double oval: two concentric ellipses, inner inset by 5px
(`INNER_BORDER_OFFSET`).

### 15.4 Cylinder Rendering

3D cylinder with top/bottom elliptical faces and a rectangular body.
The top ellipse overlaps the body slightly for the 3D effect.

### 15.5 3D Effect Rendering

For shapes with `3d: true`, render a depth offset behind the main shape:

```
THREE_DEE_OFFSET = 15px

1. Draw offset shape (shifted right and down by 15px)
2. Draw connecting side polygons (trapezoids)
3. Apply mask to hide overlap areas
4. Draw main shape on top
```

### 15.6 Multiple Effect

For `multiple: true`, render a duplicate behind with offset:

```
MULTIPLE_OFFSET = 10px

1. Draw offset duplicate (shifted right and down by 10px)
2. Draw main shape on top
```

### 15.7 Double Border

For `double-border: true`, render two concentric shapes:

```
INNER_BORDER_OFFSET = 5px

1. Draw outer shape at full size
2. Draw inner shape inset by 5px on each side
```

### 15.8 Person Shape

Stick figure above a trapezoid body:

```
      ╭───╮
      │   │     ← head (circle)
      ╰───╯
     ╱─────╲
    ╱       ╲   ← body (trapezoid)
   ╱─────────╲
       Label
```

### 15.9 Cloud Shape

Rendered as a series of overlapping circular arcs forming a cloud
outline. The cloud path uses cubic Bézier curves.

## 16. Connection Rendering

### 16.1 Path Generation

**Curved connections** (Bézier):
```
M start.x start.y
C ctrl1.x ctrl1.y ctrl2.x ctrl2.y point.x point.y
C ctrl1.x ctrl1.y ctrl2.x ctrl2.y point.x point.y
...
```

Control points are derived from route waypoints. Each pair of
consecutive points generates a cubic Bézier segment.

**Straight connections with rounded corners**:
```
M start.x start.y
L point.x point.y
S ctrl.x ctrl.y point.x point.y    ← smooth curve at corner
L point.x point.y
...
```

Corner radius is limited to `min(border_radius, segment_length / 2)`.

### 16.2 Stroke Dash Formula

```
scale = log10(-0.6 * stroke_width + 10.6) * 0.5 + 0.5
dash_size = stroke_width * dash_gap_size
gap_size = scale * dash_size
```

Applied as SVG `stroke-dasharray="{dash_size},{gap_size}"`.

### 16.3 Animated Connections

```css
.animated-connection path {
  animation: dashdraw 0.5s linear infinite;
}

@keyframes dashdraw {
  from { stroke-dashoffset: 0; }
  to { stroke-dashoffset: -dash-total; }
}
```

### 16.4 Arrowhead SVG Rendering

Arrowheads are defined as SVG `<marker>` elements:

**Triangle** (default):
```xml
<marker id="mk-triangle-..."
        markerWidth="..." markerHeight="..."
        refX="..." refY="..." orient="auto">
  <polygon points="0,0 W,H/2 0,H" fill="..." />
</marker>
```

**Arrow** (chevron):
```xml
<polygon points="0,0 W,H/2 0,H W*0.4,H/2" fill="..." />
```

**Diamond**:
```xml
<polygon points="0,H/2 W/2,0 W,H/2 W/2,H" fill="..." stroke="..." />
```

**Circle**:
```xml
<circle cx="..." cy="..." r="..." fill="..." stroke="..." />
```

**Cross**:
```xml
<polygon points="..." />  <!-- X shape -->
<path d="M ... L ..." />  <!-- connecting line -->
```

Arrowhead dimensions scale with `stroke-width`.

### 16.5 Connection Labels

Labels are positioned along the route at `label_percentage` (default 0.5
= midpoint). Text is rendered as `<text>` elements near the
computed point on the path.

## 17. Text and Label Rendering

### 17.1 Text Element

```xml
<text x="{center_x}" y="{baseline_y}"
      font-family="{family}"
      font-size="{size}"
      fill="{color}"
      text-anchor="middle">
  <tspan x="{center_x}" dy="0">Line 1</tspan>
  <tspan x="{center_x}" dy="{line_height}">Line 2</tspan>
</text>
```

### 17.2 Font Families

| Name | Usage |
|------|-------|
| Source Sans Pro | Default text font |
| Source Code Pro | Code blocks, mono shapes |

### 17.3 Text Measurement

Before layout, text dimensions are measured to size shapes:

```
shape_width  = max(min_width, text_width + 2 * padding)
shape_height = max(min_height, text_height + 2 * padding)
```

### 17.4 Label Positioning

Labels can be placed at 27 positions relative to their shape:

**Inside positions** (3x3 grid):
```
top-left     top-center     top-right
center-left  center-center  center-right
bottom-left  bottom-center  bottom-right
```

**Outside positions** (12 positions around perimeter):
```
outside-top-left    outside-top-center    outside-top-right
outside-left-top    outside-left-center   outside-left-bottom
outside-right-top   outside-right-center  outside-right-bottom
outside-bottom-left outside-bottom-center outside-bottom-right
```

**Border positions** (12 positions on the border):
```
border-top-left    border-top-center    border-top-right
border-left-top    border-left-center   border-left-bottom
border-right-top   border-right-center  border-right-bottom
border-bottom-left border-bottom-center border-bottom-right
```

### 17.5 Markdown Rendering

Markdown content is rendered via `<foreignObject>`:

```xml
<foreignObject x="..." y="..." width="..." height="...">
  <div xmlns="http://www.w3.org/1999/xhtml" class="md">
    <h1>Title</h1>
    <p>Body with <strong>bold</strong></p>
  </div>
</foreignObject>
```

### 17.6 Code Rendering

Code blocks use Chroma for syntax highlighting. Each token is rendered
as a `<tspan>` with per-token styling:

```xml
<text class="text-mono">
  <tspan fill="#keyword-color" font-weight="bold">func</tspan>
  <tspan fill="#default-color"> main() {</tspan>
  ...
</text>
```

## 18. Visual Effects

### 18.1 Shadow Filter

```xml
<filter id="shadow-filter">
  <feGaussianBlur stdDeviation="1.7" />
  <feFlood flood-color="#3d4574" flood-opacity="0.4" />
  <feComposite operator="in" />
  <feOffset dx="3" dy="5" />
  <feBlend mode="normal" />
</filter>
```

Applied to shapes (except text, code, class, sql_table) when
`shadow: true`.

### 18.2 Fill Patterns

Four SVG patterns defined in `<defs>`:

**Dots** — 15×15 tile with 9 small (1×1) rectangles at regular
intervals. Opacity 0.1, `mix-blend-mode: multiply`.

**Lines** — 15×15 tile with horizontal lines at y=2, y=7, y=12.
Opacity 0.05, `mix-blend-mode: multiply`.

**Grain** — Noise-like texture pattern.

**Paper** — Paper texture pattern.

All patterns use color `#0A0F25` with low opacity to work on any
background color.

### 18.3 Sketch Mode

When `sketch: true` in config, shapes and edges are rendered with
hand-drawn appearance (wobbly lines, imprecise corners). This uses
a rough.js-style algorithm to perturb path coordinates.

## 19. Coordinate System and Viewport

### 19.1 Coordinate Space

- Origin (0, 0) at top-left
- X increases rightward
- Y increases downward
- Units are pixels
- All positions are absolute (not relative to parent)

### 19.2 Viewport Calculation

```
DEFAULT_PADDING = 100px

viewport.x      = min(shape.x for all shapes) - padding
viewport.y      = min(shape.y for all shapes) - padding
viewport.width  = max(shape.x + shape.w) - viewport.x + padding
viewport.height = max(shape.y + shape.h) - viewport.y + padding
```

### 19.3 SVG ViewBox

```xml
<svg viewBox="{x} {y} {width} {height}"
     width="{width}" height="{height}"
     xmlns="http://www.w3.org/2000/svg">
```

## 20. Advanced Language Features

### 20.1 Variables

```d2
vars: {
  primary: "#1168bd"
  title: "My System"
}

header: ${title}
box.style.fill: ${primary}
```

### 20.2 Imports

```d2
# Import entire file (spread into current scope)
...@./shared/styles.d2

# Import specific element
my_template: @./templates/box.d2
```

### 20.3 Layers, Scenarios, Steps

Multiple views from one file:

```d2
# Base diagram (always rendered)
a -> b

layers: {
  detailed: {
    a -> b -> c -> d
  }
}

scenarios: {
  error: {
    a -> b: error! {
      style.stroke: red
    }
  }
}

steps: {
  step1: { a: highlight { style.fill: yellow } }
  step2: { b: highlight { style.fill: yellow } }
}
```

**Layers**: independent views, each a separate diagram.
**Scenarios**: inherit from base, override specific elements.
**Steps**: sequential, each step inherits from the previous.

### 20.4 Suspension

```d2
a -> b
b -> c

layers: {
  simple: {
    b -> c: { suspend }   # hide this edge in this layer
  }
}
```

`suspend` / `unsuspend` control element visibility per board.

### 20.5 Tooltips and Links

```d2
server: {
  tooltip: "Production server running Ubuntu 22.04"
  link: https://example.com/docs
}
```

Tooltips render on hover. Links make shapes clickable.

### 20.6 Icons

```d2
server: {
  icon: https://icons.terrastruct.com/essentials/server.svg
  icon-position: outside-top-center
}
```

Icons are rendered as `<image>` elements within or adjacent to shapes.
Default size 32px, max 64px.

### 20.7 Near Positioning

```d2
legend: {
  near: bottom-right
  # ...
}
```

Places elements at fixed positions relative to diagram bounds,
independent of the layout engine.
